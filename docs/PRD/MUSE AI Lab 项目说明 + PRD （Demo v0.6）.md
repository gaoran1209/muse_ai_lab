## 术语表

| 术语 | 定义 |
| --- | --- |
| 设计方案 (Project) | 最顶层容器，一个方案可包含多个 Look |
| 画布 (Canvas) | 每个设计方案对应一张无限画布，核心操作空间 |
| 素材 (Asset) | 上传的原始单品图/模特图/背景图，带标签元数据 |
| Look | 一组搭配方案（3～5 件单品的组合） |
| Board | Look 在画布中的可视化分组外框，本质是对多个对象的分组包围 |
| 节点 (Node) | 画布中的可操作单元，仅保留三类：文本、图像、视频 |
| 生成结果 (Shot) | 一次 AI 拍摄/生成的输出图或视频 |
| 内容 (Content) | 发布到 Muse Land 的最小单元（一组图 + 标题/描述/标签） |

## 0. 一句话定义

MUSE AI Lab 是面向时尚行业的 **AI 内容创作与验证平台**：创作者通过 AI 快速生成搭配内容（Muse Spark），达人通过互动和试穿验证内容价值（Muse Land），互动数据实时回流驱动下一轮创作。

*   **Muse Spark**：给内容创作者使用的"创作工作台"——选款、AI 搭配、虚拟拍摄、内容管理、数据回看。
    
*   **Muse Land**：给达人使用的"沉浸式内容场"——浏览、互动、试穿、种草、反馈偏好。

## 1. 背景与问题

### 1.1 现状痛点

*   服装内容生产链路割裂：灵感、选款、搭配、拍摄、发布、反馈分散在多个工具中。
    
*   高质量内容成本高、周期长：拍摄组织重、复用低、试错慢。
    
*   消费端反馈回流慢：达人偏好无法快速反哺到创作端。
    

### 1.2 机会点

*   用 Demo 证明：
    
    *   通过 AI 把"搭配 + 视觉生成 + 反馈分析"串成统一工作流。
        
    *   少人团队借助 AI coding 也能快速产出可演示、可迭代的时尚内容系统。
        

### 1.3 竞品定位与差异化

#### 1.3.1 竞品格局概览

市面上的 AI 时尚工具可分为四类，但没有任何一款打通"搭配创作 → AI 生成 → 消费者互动 → 数据回流"的完整链路：

| 竞品类别 | 代表产品 | 核心能力 | 缺失环节 |
| --- | --- | --- | --- |
| **通用 AI 创意画布** | TapNow | 多模型聚合生成（VEO 3.1/Kling/Midjourney）、节点式工作流、电商全链路出图 | 无搭配逻辑、无消费端、无反馈闭环 |
| **AI 电商摄影** | Claid、SellerPic、Botika、Photoroom | 平铺图→模特图、AI 换背景、批量电商图 | 只做拍摄环节，无上游搭配、无下游分发 |
| **AI 时尚设计** | Style3D、The New Black、Refabric | 草图→3D 建模、面料模拟、设计迭代 | 偏重设计研发，不做内容生产和消费端 |
| **虚拟试穿** | FASHN AI、WearView、FitRoom | 消费者端虚拟换装、体型适配 | 无创作工作台，无 B 端工具 |

#### 1.3.2 核心竞品深度分析：TapNow

TapNow（Tamar AI 旗下）是目前最接近的对标产品，定位为"AI 原生创意画布"，月访问量约 72 万（中国市场占 63%）。

**TapNow 的优势**：

*   模型聚合深度：一个平台接入 VEO 3.1、Kling、Midjourney、Imagen 等多家生成模型
    
*   Tapflow 节点式工作流：给高阶用户提供灵活的创意编排能力
    
*   电商全链路：上传 3 张产品图 + 2 个场景 → 自动生成主图/模特图/详情页，支持多平台适配
    
*   社区飞轮：TapTV 创作者社区形成内容共享和二创文化
    
*   商业化成熟：积分制（Tapies），Starter $90/年 ～ Enterprise $2160/年
    

**TapNow 的短板（= MUSE 的机会）**：

*   不理解服装搭配：缺乏品类搭配规则、风格一致性、季节适配等时尚行业知识
    
*   无消费端闭环：只有创作者社区，没有面向达人/消费者的沉浸式内容场
    
*   无反馈回流机制：内容生成后链路中断，无法用消费端偏好信号反哺下一轮创作
    
*   节点式学习曲线高：对非技术背景的设计师/买手来说，节点编排门槛较高
    

#### 1.3.3 MUSE AI Lab 的差异化定位

**一句话定位**：MUSE AI Lab 不只帮你生成内容，更帮你验证哪些内容消费者真正买单。

**两个核心壁垒**：

1.  **搭配智能**：内置服装行业的搭配知识图谱（品类关系、风格一致性、季节适配、场合匹配），AI 不只是生成图片，而是理解"什么和什么搭在一起好看"。
    
2.  **消费端闭环**：Muse Land 让内容直接触达达人，互动数据（点赞、收藏、TryOn 发起率、风格偏好）实时回流 Muse Spark，驱动下一轮创作决策——这是纯生成工具做不到的。
    

### 1.4 Demo 阶段与整体产品路线图的关系

> **背景说明**：在前期的行业研究和商业模式评估中，MUSE AI Lab 的完整愿景是「AI 视频导演 Agent」——以编排层为核心，通过多模型智能路由实现「一句话到可用商业视频」，商业模式为 SaaS + GMV 分润。

**Demo 阶段有意做了范围收窄**，从"AI 视频导演 Agent"收窄为"AI 搭配 + 生图 + 内容消费闭环"的 MVP。原因如下：

| 完整产品能力 | Demo 阶段处理方式 | 理由 |
| --- | --- | --- |
| 多模型编排 / 智能路由 | 暂时只接入单一生图 API + 单一视频 API | Demo 阶段优先验证搭配→生成→消费的核心链路，模型路由是工程优化问题，可后续叠加 |
| 视频导演 Agent（脚本→分镜→生成→剪辑） | 仅做图生视频（单镜头），不做多镜头编排 | 视频编排链路复杂度高，1 周内无法做到可用体验 |
| GMV 分润 / 电商闭环 | 不做（Demo 不含交易） | 需要商品库、订单系统，与核心验证无关 |
| 搭配智能 + 消费端闭环 | **重点验证** | **这是 MUSE 区别于 TapNow/Lovart 的核心差异化，Demo 必须证明这条路走得通** |

**Demo 如何为后续完整产品铺路**：

1.  Demo 的画布架构、Provider 模式、Task 调度系统已设计为可扩展——后续接入新模型只需实现 Provider 接口
    
2.  Demo 的节点化画布系统预留了连线传参的扩展能力，可向 TapNow 的 Tapflow 节点式工作流演进
    
3.  搭配智能的 LLM prompt 框架可逐步升级为 Agent + 知识图谱

## 2. 产品目标（Demo 阶段）

### 2.1 核心假设与验证指标

Demo 阶段最关键的任务是验证以下业务假设：

| # | 假设 | 验证方式 | 成功标准 |
| --- | --- | --- | --- |
| H1 | AI 搭配建议的准确率足以支撑创作者使用 | 搭配采纳率 | ≥ 40% 的 AI 推荐 Look 被创作者采纳或微调后采纳 |
| H2 | AI 生图质量满足时尚内容发布要求 | 生成→采纳→发布的漏斗转化率 | ≥ 30% 的生成图被采纳进入结果面板 |
| H3 | 达人愿意与 AI 生成内容互动 | Muse Land 的互动率 | 点赞/收藏率 ≥ 10%（基于内容曝光量） |
| H4 | TryOn 功能对达人有吸引力 | TryOn 发起率 | ≥ 5% 的内容浏览产生 TryOn 请求 |

### 2.2 产品目标（可验证）

*   创作者可从本地导入图片到 Muse Spark 中，完成选品和搭配
    
*   每套搭配可触发 AI 虚拟拍摄（图像为主，视频为简化版图生视频）并支持反复修改，最终采纳进入结果流
    
*   生成的内容可发布到 Muse Land，并支持点赞/收藏/评论等互动
    
*   达人可以在 Muse Land 中发起图文 TryOn，即刻查看试穿效果
    
*   Muse Spark Dashboard 展示已发布内容的互动数据概览，AI 搭配时注入近期互动偏好数据
    

### 2.3 非目标（本期不做）

*   不做完整电商交易链路（下单/支付/履约）
    
*   不做复杂组织权限体系（Demo 阶段只有 Spark 用户和 Land 用户两种入口，不区分买手/创作者角色）
    
*   不做大规模高并发和全球化架构（以 Demo 稳定性为先）
    
*   不做视频 TryOn 任务接单系统（延后到 Phase 1）
    
*   不做带货链接生成（延后到 Phase 1）
    
*   不做复杂推荐算法（Demo 内容量小，按时间倒序 + 简单标签筛选即可）

## 4. 用户旅程（User Journey）

> 本节只保留高层次的流程概览，具体功能定义详见第 5 节。

### 4.1 Muse Spark 创作流程

```plaintext
新建设计方案 → 导入素材（自动打标）→ 画布中选单品 → AI 搭配 → 浏览对比 Look
  → 确认 Look → 快捷拍摄（换模特/换背景等）→ 查看/调整/重新生成
  → 采纳 → 发布到 Muse Land
       ↓
  （可选）图生视频

```

### 4.2 Muse Land 消费流程

```plaintext
进入 Muse Land → 浏览 Feed 流 → 点赞/收藏/评论 → 打开详情（查看搭配单品）
  → 发起图文 TryOn → 上传照片 → 查看试穿结果

```

## 5. 产品设计

### 5.1 Muse Spark（创作工作台）

整体采用\*\*"三栏式 + 无限画布"\*\*的经典生产力工具布局：

*   **左侧**：素材面板（支持折叠/展开）
    
*   **中部**：核心画布区（无限画布，浅色网格背景）
    
*   **右侧**：配置与结果面板（默认隐藏，按需展开）
    
*   **顶部**：全局导航与控制栏
    

#### 5.1.1 Dashboard（工作台看板）

**布局位置**：登录后的独立首屏，点击顶部导航栏左上角 Logo 可返回。

**界面元素与交互**：

*   **新建按钮**：页面右上角显著的「+ 新建设计方案」主按钮，点击后直接进入一张全新的空白画布。
    
*   **方案列表**：以卡片网格排列历史方案。
    
    *   卡片展示：封面图（取画布内最新采纳的一张图）、方案名称、最后修改时间
        
    *   悬浮交互 (Hover)：出现"重命名"、"复制"、"删除"图标按钮；点击卡片进入该画布
        

#### 5.1.2 素材库

**布局位置**：工作台界面的左侧固定面板（支持折叠/展开）。

**界面元素与交互**：

*   **分类标签 (Tabs)**：面板上方提供 Tab 切换——\[全部\] \[商品\] \[模特\] \[背景\] \[姿势\]
    
*   **自动打标**：上传后系统自动调用 LLM API 为素材打标分类——品类（上装/下装/鞋/包/配饰）、颜色、风格、季节、场合；用户可人工修正
    
*   **素材列表区**：双列网格展示缩略图
    
    *   悬浮交互 (Hover)：右上角出现"放大镜"（预览大图）和"删除"图标
        
    *   核心交互（拖拽上屏）：鼠标按住素材图，拖至中部画布松手，即在画布对应位置生成一个图片节点
        
*   **上传规则调整（v0.6）**：
    
    *   去掉顶部「上传素材」按钮
        
    *   去掉单独的「拖拽或点击上传」区域及说明文案
        
    *   当下方列表为空时，用户可直接把图片拖到整个素材库区域完成上传
        
    *   用户也可直接把图片从本地拖到画布区域，系统同步完成上传，并在落点处生成图像对象
        

#### 5.1.3 创作画布

**布局位置**：占据屏幕中央最大面积的无限画布，深色网格背景。

##### 5.1.3.1 基础操控

*   按住鼠标左键未选中元素 → 拖拽平移画布；选中元素 → 移动元素
    
*   鼠标滚轮 → 缩放范围限制为 `50% ～ 200%`
    
*   画布支持 3 种对象节点：**文本、图片、视频**
    

##### 5.1.3.2 对象样式、选中态与分组

*   **统一对象样式（v0.6）**：所有可操作对象只保留三类，视觉结构统一参考设计图
    
    *   左上角使用文字标识对象类型：`Text｜Image｜Video`
        
    *   未选中时仅展示对象本体；单击进入选中态后，对象上方出现基础操作工具条，下方出现提示词对话框
        
    *   空状态下默认显示为空对象，不预先展示复杂参数
        
*   **文本对象**：
    
    *   只承载单一文本内容
        
    *   不再区分多种文本样式、多种文案结构或富文本块
        
*   **图像对象**：
    
    *   默认为空状态
        
    *   支持用户在下方提示词对话框输入 prompt 直接生成图像
        
    *   如果用户从本地或素材库拖入图片，该图像对象可作为后续生成链路的输入节点
        
*   **视频对象**：
    
    *   默认为空状态
        
    *   视觉上与图像对象保持一致，仅将对象类型标识改为 Video
        
*   **Board / Look 分组**：
    
    *   Look 的本质等同于 Board，本质是对多个对象的分组包围
        
    *   分组表现为多个对象外围的一圈边框，不再使用单独的大卡片承载描述内容
        
    *   一个分组可以整体被移动、删除；也允许用户单击操作分组内的某个对象
    

##### 5.1.3.3 画布空间组织逻辑

| 区域 | 功能 | 视觉特征 |
| --- | --- | --- |
| 素材暂存区 | 从素材库或本地拖入的原始图像对象，自由摆放 | 与图像对象同构，保留图像节点外观 |
| Look / Board 分组区 | AI 生成的搭配方案，每个 Look 对应一个外围分组 | 由外框包围多个对象，标题显示 Look 名称，可整体移动/删除 |
| 生成结果区 | 每个 Look 对应的生成结果图/视频 | 紧邻对应分组区域右侧，仍以图像/视频对象形式存在 |

#### 5.1.4 AI 搭配

AI 搭配建议是 MUSE 区别于通用 AI 创意工具的**核心差异化功能**。

##### 5.1.4.1 交互模式

**统一入口**：画布中选中单品后，悬浮工具条高亮显示「✨ AI 搭配」按钮。系统根据选中单品数量和品类自动判断执行策略，用户可以手动切换"补全"或"分组"。

| 系统默认行为 | 输入条件 | 输出 |
| --- | --- | --- |
| **搭配补全**：以选中单品为锚点，AI 补齐其他品类商品，生成 1～n 组完整 Look | 选中 1～2 件单品 | 每组 Look 放入画布的独立 Board 区域 |
| **搭配分组**：AI 将选中单品重新排列组合，生成多个 Look 方案 | 选中 3+ 件单品，且品类已足够丰富 | 每个组合占据一个 Board |

##### 5.1.4.2 Look 的标准构成

一个完整 Look 包含 **3～5 件单品**，标准构成为：

*   上装（必须），可单件、可内外搭配
    
*   下装（必须）
    
*   或：全身装（连衣裙等）
    
*   鞋（可选，1 双）
    
*   包/配饰（可选，0～2 件）
    

##### 5.1.4.3 LLM 输出到画布的映射规则  $\color{#0089FF}{@罗冠培(贯培)}$ 

1.  LLM 接收选中单品的标签信息（品类、颜色、风格、季节、场合），基于搭配规则 prompt 输出推荐结果
    
2.  推荐结果为结构化数据：每个推荐项包含 `品类 + 颜色 + 风格 + 搭配理由`
    
3.  系统在用户素材库中进行**标签匹配**：按品类→风格→颜色的优先级匹配
    
4.  匹配到素材 → 直接将素材图放入对应 Board 分组内，渲染为图像对象
    
5.  未匹配到素材 → 在 Board 中显示空的图像对象，占位信息通过对象的提示词对话框承载，用户可自行上传补充
    

##### 5.1.4.4 搭配过程

*   **搭配补全：**
    
    *   点击触发后，画布在所选单品旁边自动生成一个 Board 分组外框，每个 Board 对应一个 look，每个 look 由多个对象组成
        
    *   如：商品图是上衣，则调用图像生成模型生成对应裤子和鞋子的效果图
        
    *   Board 内按排版自动填入：原商品图对象 + 系统生成的单品图对象；必要时补充文本对象用于承载说明
        
*   **搭配分组：**
    
    *   点击触发后，画布对所选单品重新排列组合，生成一个或多个 Board 分组，每个 Board 对应一个 look
        
    *   Board 内按排版自动填入图像对象；如有说明信息，则通过文本对象承载
        
*   **拖拽替换**：用户从素材库拖拽新单品移到 Board 内的某张图上方时，目标图出现"替换高亮边框"，松手即完成替换，后台静默更新搭配数据
    
*   用户可在 Board 中调整后重新触发 AI 搭配推荐
    

##### 5.1.4.5 搭配逻辑（LLM Prompt 驱动）

MVP 阶段采用 LLM prompt 工程实现搭配智能。核心 prompt 包含：

*   商品图输入
    
*   品类关系规则（上下装搭配、鞋包配饰协调）
    
*   风格一致性约束（不跨风格混搭，除非明确要求混搭风）
    
*   季节和场合适配
    

#### 5.1.5 虚拟拍摄与节点化生成

拍摄系统采用**双层架构**：上层提供快捷操作入口（一键换模特/换背景），底层基于节点连线系统驱动，为后续进阶编排预留扩展能力。

##### 5.1.5.1 触发方式

选中一组确认好的 Look（Board / Group），悬浮工具条最右侧出现醒目的「📷 拍摄」主色调按钮。点击后在画布中展开生成面板 / 触发快捷节点流。

##### 5.1.5.2 虚拟拍摄

*   **配置拍摄参数：** 选中确认好的搭配组合，点击“拍摄”。在配置面板中选择预设的模特、场景、姿势、效果，并支持输入自定义文本提示词补充细节。
    
*   **任务调度与预览：** 提交任务后，后端进行排队与生成处理。生成的结果图直接展示在画布中对应搭配的旁边。用户如果不满意可调整参数重新生成。
    
*   **视频生成：** 支持文生视频或图生视频，配置面板中选择视频生成，即可出现视频节点，支持输入自定义文本提示词补充细节生视频
    

##### 5.1.5.3 快捷操作（P0 · Demo 必做）

选中 Look 后，悬浮工具条提供以下快捷生成指令：

| 快捷指令 | 交互形式 | 说明 |
| --- | --- | --- |
| **换模特** | 弹出弹窗选择预设模特（4～6 个，不同肤色/体型/性别），同时选择真人复刻还是原图换脸 | 选择后自动触发节点裂变与生成 |
| **换背景** | 弹出弹窗选择预设场景 + 支持本地上传，同时选择选择氛围融合还是选择精准替换 | 场景包含场景+姿势+效果的组合预设，如「街拍-行走」「棚拍-站姿」 |
| **TryOn** | 点击按钮 | 使用对应 prompt 进行调用 |
| **自定义指令** | 弹出文本输入框 | 兜底所有高级需求（姿势微调、氛围描述、特殊效果等） |

**A**.**快捷自动化节点流：换背景 (Change Background)** 点击 `[换背景]` 按钮，弹出气泡面板选择背景素材。气泡底部提供「处理模式」单选。系统会根据不同模式，裂变出完全不同的节点拓扑结构：

*   **模式一：✨ 氛围融合**
    
    *   **适用场景：** 提取背景图的光影与氛围，自然生成毫无违和感的大片。
        
    *   **节点裂变 (新增 3 个节点)：** 1. `背景图节点`：承载选中的背景素材。   2. `文本节点 (反推提示词)`：用于解析并提取图像特征。   3. `换背景节点`：核心处理节点。
        
    *   **连线拓扑 (Topology)：**
        
        *   `背景图节点` 连向 `文本节点` → 系统静默反推出该背景的自然语言描述（Prompt）。
            
        *   `文本节点` 连向 `换背景节点` → 将反推的 Prompt 作为生成控制条件注入。
            
        *   原 `Look 节点 (参考图)` 连向 `换背景节点` → 提供前景人物/主体的视觉参考。
            
    *   **执行逻辑：** 换背景节点综合“原图人物 + 背景反推 Prompt”，重新生成一张光影融合的新图。
        
*   **模式二：⚪️ 精准替换**
    
    *   **适用场景：** 保留背景实景的像素级原貌，仅作前景抠像替换。
        
    *   **节点裂变 (仅新增 2 个节点)：** 1. `背景图节点`   2. `换背景节点`（无文本节点，使用系统封装的预设控制指令）。
        
    *   **连线拓扑 ：**
        
        *   原 `Look 节点 (参考图)` 连向 `换背景节点`。
            
        *   `背景图节点` 直接连向 `换背景节点`。
            
    *   **执行逻辑：** 绕过文本反推，直接进行底层像素融合或硬约束重绘。
        

---

**B.快捷自动化节点流：换模特**点击 `[换模特/虚拟拍摄]` 按钮，弹出气泡面板选择模特人脸素材，并提供「处理模式」单选。根据不同模式执行不同的节点裂变：

*   **模式一：真人复刻** 
    
    *   **适用场景：** 1:1 还原指定人脸特征，结合原图服装与姿势重新生成，光影自然融合。
        
    *   **节点裂变 (新增 3 个节点)：**
        
        1.  `人脸图节点`：承载选中的目标人脸素材。
            
        2.  `文本节点 (反推提示词)`：用于解析原图构图与服装。
            
        3.  `真人复刻节点`：核心处理节点。
            
    *   **连线拓扑 (Topology)：**
        
        *   原 `Look 节点 (参考图)` 连向 `文本节点` → 系统静默反推出原图的服装、姿势、场景描述（Prompt）。
            
        *   `文本节点` 连向 `真人复刻节点` → 注入画面构图与服装内容的控制指令。
            
        *   `人脸图节点` 连向 `真人复刻节点` → 注入 Identity (面部特征) 控制。
            
    *   **执行逻辑：** 真人复刻节点使用“反推的原图场景 Prompt + 目标人脸特征”，进行全局高保真重绘。
        
*   **模式二：原图换脸** 
    
    *   **适用场景：** 像素级锁定原图的衣服与背景细节，仅强制替换面部五官。
        
    *   **节点裂变 (仅新增 2 个节点)：**
        
        1.  `人脸图节点`
            
        2.  `换脸节点`（无文本节点）。
            
    *   **连线拓扑 (Topology)：**
        
        *   原 `Look 节点 (参考图)` 直接连向 `换脸节点`（作为 Base Image）。
            
        *   `人脸图节点` 直接连向 `换脸节点`（作为 Source Face）。
            
    *   **执行逻辑：** 直接调用局部重绘或 Face Swap 算法进行面部贴合。
        

_(💡 通用交互：换模特和换背景无论哪种模式，用户选择后即刻自动执行。用户后续可拖拽新图片至对应的“输入节点”进行替换，替换后需手动点击处理节点上的 \[▶ 运行\] 按钮重新跑图。)_

**C.快捷自动化节点流：TryOn (虚拟试穿)** 点击 `[TryOn]` 按钮，触发核心的电商换装工作流。该操作会将搭配组内解构的单品，精准“穿”到指定的人物底图上。

*   **适用场景：** 验证特定服装搭配在真实模特或特定人物身上的上身效果。
    
*   **节点裂变 (新增核心节点)：**
    
    1.  `参考图节点`：承载作为底图的人物形象（即图 1）。
        
    2.  `N 个服装图节点`：系统自动读取当前搭配组，将其拆解为多个独立的服装节点（自带“上衣”、“裤子”等品类标签）。
        
    3.  `TryOn 节点`：核心换装处理节点。
        
*   **连线拓扑 (Topology)：**
    
    *   `参考图节点` 连向 `TryOn 节点`（系统默认识别其为“图 1：人物底图”）。
        
    *   搭配组拆解出的 `服装图节点` 依次连向 `TryOn 节点`（系统依次识别为“图 2”、“图 3”...）。
        
*   **执行逻辑 (Prompt 自动预置)：**
    
    *   连线完成后，`TryOn 节点` 内部会自动抓取服装连线上的品类标签，并静默拼接出一段结构化的预置 Prompt。
        
    *   **Prompt 逻辑示例：** “让图 1 的人物，穿上图 2 的 \[上衣\]，以及图 3 的 \[裤子\]。”
        
    *   系统基于该图文组合指令，调用底层的 Try-On 模型（如 VTON 算法）执行精准换装，完成后展示试穿结果图。
        

---

**D. 快捷自动化节点流：自定义指令 (Custom Instruction)** 点击 `[自定义指令]` 按钮，为创作者提供无封装的、最大自由度的生成干预口。

*   **适用场景：** 预设的换背景/换模特无法满足需求，用户需要通过自然语言直接修改画面元素（如：“改成赛博朋克画风”、“给人物戴上一副墨镜”、“将背景里的树变成红色”）。
    
*   **节点裂变 (新增 1 个节点)：**
    
    1.  `图片生成节点`：一个基础的、带有显式多行文本输入框的处理节点。
        
*   **连线拓扑 (Topology)：**
    
    *   原 `Look 节点 (参考图)` 直接连向新生成的 `图片生成节点`。
        
*   **执行逻辑：**
    
    *   系统将参考图作为垫图（Image-to-Image 或 ControlNet 约束）传入该节点。
        
    *   用户在节点的 Prompt 框中自由输入自然语言描述。
        
    *   输入完毕后，用户手动点击该节点上的 \[▶ 运行\] 按钮，大模型结合参考图与用户的自由文本指令，渲染出全新的画面结果。
        

##### 5.1.5.4 基础生成节点（P1 · 时间允许再做）

> 以下节点系统为进阶能力，用于高阶用户自由编排生成链路。Demo 阶段由快捷操作的底层自动生成，暂不开放用户手动创建。

**A. 自定义图片生成节点**

*   **连线传参（图像）**：支持接入多个图片节点，按连线先后顺序排序（图1、图2...），作为垫图输入
    
*   **连线传参（文本）**：支持接入 LLM 多模态节点，LLM 输出自动填充 Prompt 框
    
*   **面板配置**：节点表面提供自由文本 Prompt 输入框
    
*   **可选参数**：模型（默认 Nano Banana 2）、生成比例（默认 3:4）、分辨率（默认 2K）
    

**B. LLM 多模态节点**

*   **连线传参**：支持接入图片节点或其他 LLM 节点
    
*   **功能逻辑**：接收上游图像/文本进行理解分析，输出纯文本流，可连接至生图节点或视频节点作为 Prompt 控制指令
    
*   **可选参数**：模型（默认 Gemini）
    

**C. 视频生成节点**

*   **连线传参（图像限制）**：最多接入 2 个图片节点——第一根连线为首帧图 (First Frame)，第二根为尾帧图 (Last Frame)
    
*   **连线传参（文本）**：支持接入 LLM 节点，用于视频运镜或动态 Prompt
    
*   **面板配置**：文本 Prompt 输入框
    
*   **可选参数**：模型（默认可灵 Kling）、分辨率（默认 2K）
    

##### 5.1.5.5 视频策略

Demo 阶段仅做**图生视频**：从已生成的效果图出发，调用视频生成 API 生成短视频。视频节点展示在画布中，支持自定义 prompt 补充动态描述。

##### 5.1.5.6 生成任务流程

```plaintext
用户触发生成（快捷操作 / 手动运行节点）
  → 创建 GenerationTask（状态：排队）
  → 后端调度生成（状态：处理中），节点显示进度条
  → 生成完成，结果图展示在节点/结果区（状态：完成）/ 失败重试（状态：失败）

```

*   每次拍摄生成 **1 张** 结果图，输出展示在画布中（紧邻对应 Look Board）
    
*   用户不满意可调整参数后重新生成（生成新任务，不覆盖旧结果）
    
*   用户在画布中可查看本次 Look 的所有历史生成结果
    

#### 5.1.6 效果定稿

##### 5.1.6.1 "采纳"操作定义

*   用户对画布中的某张生成图/视频点击「✅ 采纳」按钮
    
*   采纳后：该内容以缩略图形式"飞入"右侧**结果面板**，标记为"已采纳"状态
    
*   采纳后图片**同时保留在画布中**，用户可继续编辑
    
*   用户可取消采纳（悬浮卡片点击"取消采纳"），不影响画布中的内容
    

##### 5.1.6.2 结果面板

**布局位置**：默认隐藏，通过点击顶部导航栏右侧的「📦 结果箱 / 已采纳」按钮切换至右侧面板显示。

结果面板**按 Look 分组展示**已采纳的图片和视频：

*   每个 Look 为一个折叠组，展示该 Look 下所有已采纳的 Shot
    
*   组内支持拖拽排序
    
*   每个 Shot 显示缩略图 + 采纳时间
    

##### 5.1.6.3 发布操作

*   用户在结果面板中选中一个或多个 Shot → 点击「发布到 Muse Land」（面板底部有「批量发布」主按钮）
    
*   弹出发布编辑框：
    
    *   标题（支持 AI 自动生成，基于搭配的风格标签）
        
    *   描述文案（支持 AI 自动生成）
        
    *   标签（自动从 Look 的风格标签继承，可编辑）
        
*   确认发布 → 数据写入数据库 → Muse Land Feed 流实时展示
    
*   飘字提示"发布成功"
    
*   Demo 阶段无需审核

## 7. 全流程验证标准（Demo 必通）

### 7.1 创作与打包 (Muse Spark 端)

*   **交互动作**：创作者上传核心单品图 → 拖入画布 → 选中后点击「AI 搭配」→ 系统在画布上自动补齐其他服饰形成完整 Look（Board）→ 点击「拍摄」→ 选择模特/背景 → 节点裂变并自动生图
    
*   **必通标准**：生成的结果图能一键「采纳」，在结果面板中按 Look 分组展示，并绑定该 Look 的所有单品信息形成完整数据包
    

### 7.2 内容分发 (Spark → Land)

*   **交互动作**：创作者在结果面板中选中已采纳图片 → 编辑标题/描述 → 点击「发布到 Muse Land」
    
*   **必通标准**：数据无缝写入数据库。刷新 Muse Land Feed 流，内容立即出现，点开详情能完整看到搭配单品列表（Demo 阶段无需审核）
    

### 7.3 TryOn 试穿 (Land 端)

*   **交互动作**：达人在 Muse Land 浏览到内容 → 点击「TryOn 试穿」→ 上传正面照 → 提交
    
*   **必通标准**：系统异步生成试穿效果图，达人能在页面中查看结果

## 8. Vibe Coding 协作流程

> 本节面向团队中非技术背景的 PM/设计师，说明如何与 AI 编程工具协作推进开发。

### 8.1 前后端项目结构速查

项目主要分为两大块：

#### 前端区 (`src/frontend/src/`)

*   `pages/`：路由级页面组件（如首页 Home、画布页 Canvas）。加新页面在这里。
    
*   `components/`：页面内积木（按钮、弹窗、面板等）。改界面元素在这里。
    
*   `store.ts`：Zustand 状态管理（用户选了什么模型、输入了什么 prompt 等）。
    
*   `hooks/`：自定义 React Hooks（如 `useFabricCanvas.ts` 封装 Fabric.js）。
    

#### 后端区 (`src/backend/`)

*   `api/`：前后端通信接口。
    
*   `services/`：核心业务逻辑（图片处理、工作流管理等）。
    
*   `providers/`：**AI 模型供应商目录**
    

### 8.2 标准操作流程

#### 先 Plan 后执行

使用 AI 编程工具时，先用以下模板让 AI 输出修改计划，确认无误后再执行：

> "我是一个产品经理，不懂具体的代码逻辑。现在需要在当前项目中开发一个新功能：**\[描述你的功能\]**。请你基于我们现有的 `muse_studio/` 目录结构，帮我梳理开发方案，**用一张表格**向我汇报你的修改计划，表格包含以下三列：1. 要修改的文件路径；2. 产品视角的修改目的（用人话解释）；3. 风险等级（高/中/低）。**注意目前是列计划，请只给方案列表，不要写代码。**"

#### 修 Bug 专属模板

> "我在测试时遇到了一个 Bug。**场景：**\[在哪个界面\]。**操作与预期：**\[我点了什么，期望发生什么\]。**实际表现：**\[实际发生了什么\]。控制台报错：`[粘贴报错信息]`。请帮我定位问题并直接给出修复代码，修复范围越小越好，不要大规模重构。"
